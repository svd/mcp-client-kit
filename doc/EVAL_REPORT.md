## Overall Verdict

The `generate-mcp-wrappers` skill performed well across all 13 servers: every run produced a passing verdict (all 5 verifier checks — ast, signatures, idempotency, pii, roundtrip — passed or were correctly skipped), and no run required manual intervention to fix generated code. Mode selection was accurate in all observed cases, including the correct principled deferral to `-> Any` on schema-unstable tools (sqlite `read_query`, git paths, `get-env`).

## Mode Coverage Gaps

Ten of 13 servers landed entirely in Mode A (unstructured string). For most this is correct — codegraph, context7, deepwiki, fetch, git, huggingface, and codegraph genuinely return Markdown prose. However three cases merit a re-probe:

- **deepwiki `read_wiki_structure`**: returns a separator-delimited listing that could be parsed into a `List[str]` (Mode C) if the delimiter is stable. A second probe with a different repo would confirm.
- **everything** (8 of 9 probed tools): many tools return structured content inside the MCP envelope rather than a parsed dict; if the envelope were unwrapped one level, several could yield TypedDicts (Mode B). The `get-structured-content` tool correctly reached Mode B, which suggests the others may be envelope-blocked candidates.
- **sqlite `read_query`**: correctly left as Mode A because output shape is query-dependent, but a discriminator on a small, fixed query set (e.g., `sqlite_master`) could in principle yield a Mode C list.

## Known Issues

1. **Roundtrip systematically skipped on Mode-A servers**: 8 of 13 servers skipped roundtrip with `no_shaped_non_mutating_tool`. This is correct per the verifier contract but means live-call validation is absent for the majority of the corpus. Any regression in the `-> Any` path would not be caught.
2. **`signatures` check skipped on github**: The `get_team_members` tool returned `NoneType` due to PAT scope limits, causing `probe_inconclusive` and a signatures skip. The verifier does not distinguish "probe failed due to auth" from "tool genuinely returns null" — both are opaque.
3. **`branches` unobservable in sequential-thinking**: The `ThoughtResult.branches` field stayed `list` (element type unknown) because the single-step probe returned an empty list. The shapes.json annotates this gap, but no automated re-probe is triggered.

## Next Steps

1. **Add envelope-unwrap detection as a Pass-0 step**: Before mode assignment, attempt one-level JSON unwrap on all string-returning tools to surface hidden structure (addresses the `everything` gap and would increase Mode B/C coverage).
2. **Distinguish auth-blocked probes from structural nulls**: Add an `auth_error` probe status so the verifier can skip `signatures` cleanly without conflating access failure with type evidence — fixes the github `probe_inconclusive` ambiguity.
3. **Automate re-probe for empty-list fields**: When a shaped field is typed `list` with unknown element type, schedule a follow-up probe with inputs that are likely to produce non-empty results (e.g., sequential-thinking with `totalThoughts > 1`), and merge the inferred element type back into shapes.json.
